Remote

How to bind a remote repository
to my local one?

Binding to "remote" repository could be handled with command remote

Subcommand add allows to create a new reference to a remote repository

git remote add <ref_name> <uri_to_repo>

add requires an arbitrary name for the reference and an uri to the "remote" repo

n.b. git supports https, ssh or path
Remote 
git remote add origin https://gitlab.org/path/repository.git

What has changed?

git remote -v
origin  https://gitlab.org/path/repository.git (fetch)
origin  https://gitlab.org/path/repository.git (push)

Two references to a remote repository
(which I called "origin") were created,
one to send commits (push)
and one to receive commits (fetch)

Remote

How to "update" a remote repository?

git push <remote_ref_name> <ref>

Command push takes a reference to a remote and a reference to commit, then it sends to the remote all the commits which reference depends on

The reference to commit can be a tag or a branch(branch's head)
i.e. a pointer to a commit (refs in official guide)
Remote 
git push origin master

git sends the commits on which the branch master depends
to the remote repository called origin,
which tries to merge them to its repository's history.

n.b. if there are conflict between remote and local commits, the remote will refuse to accept the push ️merging conflict have to be solved locally before push ⚠️
Remote

And what if we want to download
the remote commits?

git pull <remote_ref_name> <ref>

Command pull tries to retrive a ref(and associated commits) from a remote and merging with the local history

Remote 
git pull origin master

git tries to download the commits of branch master
from the remote repository called origin,
and it merges them with the local history.

Remote

It tries... and what if is there a conflict?

Guess what...

...we have to correct it by ourselves!

In a way very similar to how we have seen
in the branch section.

Remote

It tries... and what if is there a conflict?

Also for a conflict during a push operation
we have to manually handle the conflict

Conflicts must always be resolved
by the client-side!

We have to do a pull operation,
solve the conflicts and then do again the push operation.

As a best practice, do a pull operation ever before resuming to work over a project.
Remote

If we want to download a whole remote repo...

We can do it with the previous commands

mkdir <repo_name>
cd <repo_name>
git init
git remote add origin <repo_uri>
git pull

...or, we can also use command clone

git clone <repo_uri>
